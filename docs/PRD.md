## 1. 项目概述 (Project Overview)

### 1.1 背景
本项目旨在解决从大众点评 SaaS 平台下载的业务数据无法直接对接服务端的问题。目标是开发一个“黑箱”客户端工具，能够自动清洗脏数据、去重、按特定规则分片上传，并支持断点续传。

### 1.2 核心目标
*   **智能化**：零配置，拖拽即用，自动识别表头。
*   **稳定性**：**业务主键去重**，容错率高（失败跳过、连续熔断）。
*   **规范性**：符合服务端分片要求，严格遵循交互原则。
*   **易用性**：原生应用体验，无环境依赖。

```mermaid
graph TD
    User((用户)) -->|拖拽 Excel 文件| UI[客户端界面]
    UI -->|读取文件| Parser[智能解析引擎]
    
    subgraph "ETL 处理核心 (黑箱)"
        Parser -->|1. 识别有效表头| HeaderCheck{表头校验}
        HeaderCheck -->|匹配成功| Cleaner[数据清洗 & 日期转码]
        HeaderCheck -->|匹配失败| Error[报错提示]
        
        Cleaner -->|2. 生成业务指纹| Fingerprint{生成指纹}
        Fingerprint -->|基于业务ID/组合键| Deduplicate{本地去重}
        
        Deduplicate -->|指纹不存在| DB[(本地 SQLite 中间库)]
        Deduplicate -->|指纹已存在| Skip[跳过/更新状态]
    end
    
    subgraph "传输调度 (Uploader)"
        DB -->|查询待上传 (PENDING)| Scheduler[调度器]
        Scheduler -->|1. 按门店名称分组| Grouping
        Grouping -->|2. 按时间升序排序| Sorting
        Sorting -->|3. 分片 (100条/包)| Batching[构建 Batch JSON]
        
        Batching -->|POST 请求| API[服务端接口]
        
        API -->|返回 200 OK & code=0| Success[标记 Success]
        API -->|返回 错误/超时| Retry[进入重试流程]
        
        Retry -->|重试3次失败| SkipBatch[标记 SKIPPED & 继续下一个]
        SkipBatch -->|连续5个Batch失败| Fuse[触发熔断 & 暂停队列]
        
        Success -->|更新进度| UI
    end
```

## 3. 详细功能需求 (Functional Requirements)

### 3.1 输入与解析 (Input & Parsing)

#### 3.1.1 文件支持
*   支持 `.xlsx`，支持批量拖拽。
*   **体验优化**：文件处理完成后，自动重命名文件（增加 `_processed` 后缀）或移动至 `processed` 文件夹，避免重复操作。

#### 3.1.2 表头行指定与处理
*   **表头行直接指定**：根据文件类型，直接使用指定的表头行，无需自动识别。
    *   **会员交易明细**：Row 2（1-based），即 Row 1（0-based）
    *   **店内订单明细**：Row 3（1-based），即 Row 2（0-based）
    *   **收入优惠统计**：Row 3（1-based），即 Row 2（0-based）
    *   **优惠券统计表**：Row 3（1-based），即 Row 2（0-based）
    *   **会员储值消费分析表**：Row 4（1-based），即 Row 3（0-based）
    *   **会员卡导出**：Row 2（1-based），即 Row 1（0-based）
*   **表头拆分与填充**：
    *   **拆分合并单元格**：对于表头行中的 None 值，检查是否为合并单元格的一部分。如果是，获取合并区域的左上角单元格值进行填充。
    *   **填充跨行表头**：对于拆分后仍为 None 的位置，检查下一行（Row N+1）是否有值。如果有，使用下一行的值填充；如果当前行和下一行都有值，进行拼接合并（如：`累计储值金额（元）储值余额累计（元）`）。
    *   **移除末尾空列**：从右向左扫描，移除所有连续的 None 列（这些列在数据中不存在）。
*   **有效数据**：填充后的表头行的下一行起，至"合计/总计"行或空行止。

#### 3.1.3 数据清洗规则 (Strict Cleaning)

##### 3.1.3.1 全文本策略
*   解析器**强制**将 Excel 所有单元格读取为 **String** 类型，严禁自动转换为 Number。

##### 3.1.3.2 特殊占位符处理
*   **占位符列表及处理**：
    *   `"--"` → 转换为空字符串 `""`
    *   `"-"` → 转换为空字符串 `""`
    *   `"NULL"`、`"null"`、`"Null"` → 转换为空字符串 `""`
    *   `"N/A"`、`"n/a"`、`"NA"` → 转换为空字符串 `""`
*   **处理时机**：在字符串规范化阶段统一处理。
*   **业务场景**：
    *   `退单时间` 字段可能包含 `"--"`（表示未退单）
    *   `券核销张数` 字段可能包含 `"--"`（表示不适用）
    *   `储值消费金额` 字段可能包含 `"-"`（表示无消费）

##### 3.1.3.3 空值与零值的严格区分 (CRITICAL)
*   **场景 A (零值)**：Excel 单元格内容显式为 `0`、`0.00` 或 `"0"` (文本)。
    *   **处理**：必须保留为字符串 `"0"` 或 `"0.00"`（根据 number_format 保留小数位）。
    *   *业务含义：发生了业务，但数值为 0（如积分抵扣、赠品）。*
*   **场景 B (空值)**：Excel 单元格为空白 (Empty / Null / Undefined) 或特殊占位符。
    *   **处理**：必须保留为 **空字符串** `""`。
    *   *业务含义：该字段不适用或无记录。*
*   **严禁操作**：
    *   禁止使用类似 `if (!value) value = 0` 的逻辑。
    *   **禁止将空值默认填补为 0**。
    *   空值必须保留为空字符串，零值必须保留为字符串形式的 "0"。

##### 3.1.3.4 数据类型转换规则
*   **数字类型处理**：
    *   `int` → 转换为字符串，保留原值（如 `123` → `"123"`）
    *   `float` → 转换为字符串：
        *   如果有 `number_format`，按格式保留小数位（如 `0.00` → `"0.00"`，`12.30` → `"12.30"`）
        *   否则，去除尾随零（如 `12.30` → `"12.3"`，`12.00` → `"12"`）
        *   但 `0.00` 格式必须保留为 `"0.00"`（如果 number_format 指定了小数位）
*   **布尔类型处理**：
    *   `True` → `"TRUE"`
    *   `False` → `"FALSE"`
*   **百分比处理**：
    *   所有百分比字段都转换为小数字符串：`"25%"` → `"0.25"`，`"6.45%"` → `"0.0645"`，`"100%"` → `"1.0"`
    *   转换规则：移除 `%` 符号，将数值除以 100 得到小数
    *   适用于所有文件类型中的所有百分比字段
*   **日期特殊处理**：
    *   针对所有包含"时间"、"日期"字样的列。
    *   **Excel 序列号**（如 `45231.5`）→ 转换为 `YYYY-MM-DD HH:mm:ss`
    *   **datetime 对象** → 转换为 `YYYY-MM-DD HH:mm:ss`
    *   **字符串日期**：
        *   尝试解析常见格式：`YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY-MM-DD HH:mm:ss` 等
        *   解析成功 → 标准化为 `YYYY-MM-DD HH:mm:ss`
        *   解析失败 → 保留原字符串
    *   **空值** → 置为空字符串 `""`

### 3.2 本地存储与去重 (Local Storage & Deduplication)

#### 3.2.1 存储方案
使用 **SQLite** 嵌入式数据库。

#### 3.2.2 唯一性保障 (业务去重)
废弃全字段 MD5，采用**业务逻辑主键**生成 `fingerprint`：

*   **明细类表**：直接使用原始数据的唯一 ID 列。
    *   《会员交易明细》 -> `交易流水号`（若为空，退化为整行 MD5）
    *   《店内订单明细》 -> 优先 `订单明细表`（第一列 ID）；若不存在，退化为 MD5(`门店`+`订单号`+`下单时间`)；再退化为整行 MD5
    *   《会员卡导出》 -> `会员卡号`（若为空，退化为整行 MD5）
*   **统计类表**：使用维度组合生成的字符串 (MD5)。
    *   《收入优惠统计》 -> MD5(`门店` + `营业日期` + `编码` + `结账方式类型` + `结账方式` + `类型`)（若组合为空，退化为整行 MD5）
    *   《优惠券统计表》 -> MD5(`交易日期` + `门店` + `券名称`)（若组合为空，退化为整行 MD5）
    *   《会员储值消费分析表》 -> MD5(`交易日期` + `机构编码` + `卡类型名称`)（若组合为空，退化为整行 MD5）

**去重机制说明**：
*   `upload_tasks.fingerprint` 字段在 SQLite 中设置为 **UNIQUE 唯一索引**。
*   当解析 Excel 时，若 `fingerprint` **已存在**，则插入失败（IntegrityError），该行被计为**重复**，**不会覆盖原有记录的状态**。
*   只有 `fingerprint` **不存在**的行才会插入，状态初始化为 `PENDING`（等待推送）。
*   这意味着：**同一份 Excel 重复解析时，已成功推送的数据不会再次进入队列**，保证数据一致性。

### 3.3 传输调度 (Transmission Scheduler)

#### 3.3.1 分组与排序
*   **一级分组**：
    *   **普通数据源**：优先按 `storeId`（门店机构编码）分组；若 `storeId` 为空或为 `-`，则按 `storeName` 分组。
        *   **会员交易明细**：优先使用 `操作门店机构编码`（兜底：`开卡门店机构编码`）。
        *   **店内订单明细**：使用 `机构编码`。
        *   **收入优惠统计**：使用 `机构编码`（通常无，返回空）。
        *   **优惠券统计表**：使用 `机构编码`。
        *   **会员储值消费分析表**：使用 `机构编码`。
        *   **空店处理**：若 `storeId` 为空，统一归为"空店"组，`storeName` 统一显示为 `"空"`，保证数据一致性。
    *   **会员卡导出**：按 `卡等级` 列分组，参数名为 `level`。若 `卡等级` 为空，统一归为"空等级"组。
*   **二级排序**：
    *   **普通数据源**：按 `timestamp` ASC（时间升序）。
        *   **会员交易明细**：按 `交易时间` 升序。
        *   **店内订单明细**：按 `下单时间` 升序。
        *   **收入优惠统计**：按 `营业日期` 升序。
        *   **优惠券统计表**：按 `交易日期` 升序。
        *   **会员储值消费分析表**：按 `交易日期` 升序。
    *   **会员卡导出**：按 `开卡门店` 升序。

#### 3.3.2 分片上传
*   **Batch Size**: 100 条。
*   **Payload 结构**：
    *   **普通数据源**：
        ```json
        {
          "platformKey": "...",
          "storeId": "MD00005",  // 必填：若为空则传空字符串 ""
          "storeName": "山禾田·日料小屋（龙华店）",  // 必填：若 storeId 为空，则统一为 "空"
          "data": [ ... ]
        }
        ```
        *   **空店处理**：若 `storeId` 为空，payload 中 `storeId` 传空字符串 `""`，`storeName` 统一传 `"空"`，确保所有无门店编码的数据能统一推送，保持数据一致性。
    *   **会员卡导出**：
        ```json
        {
          "platformKey": "...",
          "level": "VIP1",  // 必填：卡等级，若为空则传 "空等级"
          "data": [ ... ]
        }
        ```
        *   不包含 `storeId` 和 `storeName` 字段。
        *   若 `卡等级` 为空，`level` 传 `"空等级"`。

#### 3.3.3 容错机制 (Resilience)
*   **单包重试**：Batch 失败 -> 等待 `2s` -> 重试 1 -> `5s` -> 重试 2 -> `10s` -> 重试 3。
*   **严格顺序推送 (CRITICAL)**：
    *   若某一 Batch **未获得任何响应**（网络超时/连接失败，`status_code` 为 `None`），在完成所有重试后，**立即停止后续所有 Batch 的推送**，抛出 `NoResponseStopError`。
    *   这是为了防止在网络异常时继续发送无效请求，确保数据推送的可靠性。
*   **失败跳过 (Skip)**：
    *   若 Batch 获得响应但业务失败（HTTP 200 但 `code != 0`，或 HTTP 非 200），3 次重试均失败后，标记该 Batch 为 `SKIPPED`，记录错误日志。
    *   **继续执行下一个 Batch**（仅在获得响应的情况下）。
*   **连续熔断 (Circuit Breaker)**：
    *   若检测到**连续 5 个** Batch 最终状态均为 `SKIPPED`/`FAILED`。
    *   触发熔断：暂停该门店的所有后续任务，UI 弹窗报警，防止无效请求刷屏。
*   **自动重推机制**：
    *   在每次推送开始前，自动执行：
        1.   **回填 storeId**：对于 `PENDING` 状态但 `store_id` 为空的记录，从 `raw_data` 中提取 `storeId` 并更新。
        2.   **重推已跳过数据**：对于 `会员交易明细` 中 `SKIPPED` 状态但 `store_id` 为空的记录，若 `raw_data` 中存在门店编码，则回填 `store_id` 并将状态重置为 `PENDING`，允许重新推送。

### 3.4 交互设计 (UI/UX) - Python Native GUI

遵循“简单、实用、稳定”原则，采用原生 GUI 框架开发。

#### 3.4.1 技术选型与视觉
*   **框架**：使用 **Tkinter** + **ttkbootstrap**，确保在 Windows/Mac 下有一致的原生体验。
*   **拖拽支持**：优先使用 `tkinterdnd2` 实现拖拽功能；若系统不支持（如 `tkdnd` 库未安装），自动降级为“选择文件”按钮。
*   **窗口**：默认尺寸 `520px * 760px`，**允许用户调整大小**（`resizable=True`）。
*   **风格**：简洁的商务风格。
    *   主色调：浅灰/白。
    *   强调色：**点评橙 (#FF6633)** (用于按钮、进度条)。

#### 3.4.2 核心页面布局
1.  **主界面 (Main Window)**
    *   **顶部区域**：
        *   **运行模式横幅**（最重要）：显眼显示当前推送模式："预演（不发送请求）" 或 "真实推送（发送 webhook）"，确保用户明确当前操作。
        *   标题 "Dianping ETL Uploader"。
    *   **控制区域**：
        *   **推送模式选择**：单选框组，选项为：
            *   "预演（不发送请求）"：仅预览，不发送 HTTP 请求，不更新数据库状态。
            *   "真实推送（发送 webhook）"：真实发送 HTTP 请求，更新数据库状态。
        *   **配置输入**：平台 Key 输入框、归档设置等。
    *   **操作按钮区域**：
        *   **开始处理**：解析 Excel 文件，清洗、去重、入库（不推送）。
        *   **仅推送待上传**：仅推送数据库中 `PENDING` 状态的数据，不解析文件。
        *   **停止**：中断当前任务。
        *   **重置**：清空数据库队列 + 解除熔断状态。
    *   **统计面板**（本次结果，不用看日志）：
        *   **解析统计**：显示本次解析的行数、插入数、重复数、跳过数。
        *   **推送统计**：显示本次推送的批次、成功数、跳过数、停止原因。
    *   **进度条**：显示当前任务进度。
    *   **日志框**：只读的 `ScrolledText` 组件，实时打印带时间戳的日志。
    *   **状态栏**：显示当前状态（如 "就绪"、"正在推送..."）。

2.  **状态反馈**
    *   **成功**：状态栏显示 "推送完成"，统计面板更新。
    *   **异常**：日志框自动滚动到最新报错行，状态栏显示错误信息。
    *   **确认对话框**：在"真实推送"模式下，点击"仅推送待上传"时会弹出确认对话框，防止误操作。

#### 3.4.3 交互逻辑
*   **防冻结**：所有耗时的 ETL 和上传任务必须放在 **独立线程 (`threading.Thread`)** 中运行，严禁阻塞主 UI 线程，确保界面在上传时依然响应拖拽和点击。
*   **配置持久化**：用户设置的 `platformKey`、推送模式、归档设置、最后打开的目录会自动保存到本地 JSON 文件，下次启动时自动加载。
*   **熔断恢复**：提供一个显眼的"重置"按钮，用于清除数据库队列和熔断状态。
*   **模式同步**：推送模式通过 `StringVar.trace_add` 实时同步到配置对象，确保 UI 显示与实际行为一致。

## 4. 接口规范 (API Specification)

### 4.1 请求结构 (Updated)
```json
{
  "platformKey": "f5edd587da...",
  "storeId": "MD00001",           // 必填：若 Excel 里有编码则传编码，若无则传空字符串 ""
  "storeName": "山禾田·日料小屋（龙华店）", // 必填：若 storeId 为空，则统一传 "空"
  "data": [ ... ]                 // 数组，每项为一条清洗后的数据记录，包含 fingerprint 字段
}
```

**字段说明**：
*   `storeId`：**必填字段**。优先使用 Excel 中的门店机构编码（如 `操作门店机构编码`、`机构编码`）。若为空或为 `-`，则传空字符串 `""`。
*   `storeName`：**必填字段**。若 `storeId` 为空，则统一传 `"空"`，保证数据一致性；否则传 Excel 中的门店名称。
*   `data`：数组，包含该批次的所有数据记录（最多 100 条）。每条记录包含：
    *   所有清洗后的原始数据字段
    *   `fingerprint`：**唯一性标识字段**，用于服务端判断数据唯一性
        *   明细类表：使用原始唯一ID（如 `交易流水号`、`订单明细表`、`会员卡号`）
        *   统计类表：使用 MD5 组合键（如 `MD5(门店+营业日期+编码+结账方式类型+结账方式+类型)`）

**服务端职责**：
*   根据 `storeId` (优先) 或 `storeName` 匹配真实门店 ID。匹配失败则记录异常，但不阻断流程。
*   使用每条记录中的 `fingerprint` 字段判断数据唯一性，避免重复处理。

### 5.2 各数据源详细处理规则

#### 5.2.1 会员交易明细
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `交易时间` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
*   **去重标识 (Fingerprint)**：
    *   优先使用 `交易流水号` 作为唯一标识。
    *   若 `交易流水号` 为空，退化为整行 MD5。
*   **排序标识 (Timestamp)**：
    *   使用 `交易时间` 字段进行时间升序排序。
*   **门店编码提取 (storeId)**：
    *   优先提取 `操作门店机构编码`。
    *   若为空，兜底使用 `开卡门店机构编码`。
    *   若仍为空，返回空字符串。

#### 5.2.2 店内订单明细(已结账)
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `下单时间`、`结账时间` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
*   **去重标识 (Fingerprint)**：
    *   优先使用 `订单明细表`（第一列 ID）作为唯一标识。
    *   若不存在，退化为 MD5(`门店` + `订单号` + `下单时间`)。
    *   若组合仍为空，再退化为整行 MD5。
*   **排序标识 (Timestamp)**：
    *   使用 `下单时间` 字段进行时间升序排序。
*   **门店编码提取 (storeId)**：
    *   提取 `机构编码`。
    *   若为空，返回空字符串。

#### 5.2.3 收入优惠统计
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `营业日期` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
*   **去重标识 (Fingerprint)**：
    *   使用 MD5(`门店` + `营业日期` + `编码` + `结账方式类型` + `结账方式` + `类型`) 作为唯一标识。
    *   若组合为空，退化为整行 MD5。
*   **排序标识 (Timestamp)**：
    *   使用 `营业日期` 字段进行时间升序排序。
*   **门店编码提取 (storeId)**：
    *   提取 `机构编码`（通常无，返回空字符串）。

#### 5.2.4 优惠券统计表
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `交易日期` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
*   **去重标识 (Fingerprint)**：
    *   使用 MD5(`交易日期` + `门店` + `券名称`) 作为唯一标识。
    *   若组合为空，退化为整行 MD5。
*   **排序标识 (Timestamp)**：
    *   使用 `交易日期` 字段进行时间升序排序。
*   **门店编码提取 (storeId)**：
    *   提取 `机构编码`。
    *   若为空，返回空字符串。

#### 5.2.5 会员储值消费分析表
*   **表头处理**：
    *   **主表头行**：Row 4（1-based），即 Row 3（0-based）
    *   **跨行表头合并**：需要与 Row 5（1-based）合并处理。
        *   若 Row 4 和 Row 5 同列都有值，则合并为 `Row4值Row5值`（如：`累计储值金额（元）储值余额累计（元）`）
        *   若 Row 4 为 None，Row 5 有值，则使用 Row 5 的值
        *   若 Row 4 有值，Row 5 为 None，则保留 Row 4 的值
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `交易日期` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
    *   **百分比字段**：所有百分比字段（如 `储值会员占比`、`未消费储值占比` 等）都需要转成小数字符串，例如 `25%` → `0.25`，`6.45%` → `0.0645`。
    *   **数字字段**：大量 float 类型字段（如 `会员数量`、`储值会员数`、`累计储值金额（元）` 等），需根据 number_format 正确转换。
*   **去重标识 (Fingerprint)**：
    *   使用 MD5(`交易日期` + `机构编码` + `卡类型名称`) 作为唯一标识。
    *   若组合为空，退化为整行 MD5。
*   **排序标识 (Timestamp)**：
    *   使用 `交易日期` 字段进行时间升序排序。
*   **门店编码提取 (storeId)**：
    *   提取 `机构编码`。
    *   若为空，返回空字符串。

#### 5.2.6 会员卡导出
*   **表头处理**：
    *   **主表头行**：Row 2（1-based），即 Row 1（0-based）
    *   表头字段包括：姓名、会员卡号、会员手机号、卡类型、卡等级、开卡时间、开卡门店等
*   **数据清洗**：
    *   所有字段强制转换为字符串类型。
    *   空值与零值严格区分：空值保留为空字符串 `""`，零值保留为 `"0"` 或 `"0.00"`。
    *   `开卡时间` 列：若为 Excel 序列号，转换为 `YYYY-MM-DD HH:mm:ss` 格式；若为空，置为空字符串。
    *   **数字字段**：如 `卡押金（元）`、`储值余额合计（元）`、`积分余额` 等，需根据 number_format 正确转换。
*   **去重标识 (Fingerprint)**：
    *   使用 `会员卡号` 作为唯一标识。
    *   若 `会员卡号` 为空，退化为整行 MD5。
*   **分组规则**：
    *   **一级分组**：按 `卡等级` 列分组，参数名为 `level`。
    *   若 `卡等级` 为空，统一归为"空等级"组。
*   **排序规则**：
    *   **二级排序**：按 `开卡门店` 升序排序（不是按开卡时间）。
*   **门店编码提取 (storeId)**：
    *   提取 `机构编码`（通常无，返回空字符串）。
    *   **注意**：Payload 中不包含 `storeId` 字段。
*   **门店名称提取 (storeName)**：
    *   优先使用 `开卡门店`，若为空则使用 `首次交易门店` 或 `招募门店`。
    *   **注意**：Payload 中不包含 `storeName` 字段。

**状态管理**：
*   `PENDING`：待推送（新插入的数据，或从 `SKIPPED` 重置为 `PENDING` 的数据）。
*   `SUCCESS`：推送成功。
*   `SKIPPED`：推送失败（3 次重试后仍失败，或熔断触发）。
*   `FAILED`：推送失败（保留用于未来扩展）。

*注：对于统计表，尽量选取能唯一确定一行的维度组合。*

## 6. 运行与交付 (Run & Delivery)

本项目以源码脚本形式交付，直接运行 Python 入口文件启动。

### 6.1 运行环境要求
*   **Python 版本**：Python 3.10+

    *   `python-dateutil`：日期解析

*   提供启动入口 `main.py`。
*   用户可在终端运行 `python main.py` 调起 GUI 界面。
*   数据库文件：默认在当前工作目录创建 `siyu_etl.sqlite3`。
*   配置文件：默认在当前工作目录创建 `.siyu_etl_config.json`（存储用户设置）。

### 6.3 自动更新 (不适用)
*   由于采用源码运行，更新方式为拉取最新 Git 代码或替换源码文件。
