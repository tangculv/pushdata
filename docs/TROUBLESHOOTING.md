## 常见错误

### 1. 文件被占用错误

**错误信息**：
```
无法读取文件 xxx.xlsx：文件可能正在被其他程序（如 Excel）打开。
请关闭文件后重试。原始错误: ...
```

**原因**：
- Excel 文件正在被其他程序打开（如 Microsoft Excel、WPS）
- 文件被其他进程锁定

**解决方法**：
1. 关闭所有打开该文件的程序
2. 检查任务管理器/活动监视器，确保没有相关进程
3. 如果问题持续，尝试复制文件到新位置再处理
4. 重新尝试操作

**预防措施**：
- 处理文件前确保文件已关闭
- 避免在处理过程中打开文件

---

### 2. 无法识别文件类型

**错误信息**：
```
无法识别表类型: xxx.xlsx
```

**原因**：
- 文件名不包含识别关键词
- 文件不是支持的 6 种类型之一
- 文件格式不正确

**解决方法**：
1. 确认文件是支持的 6 种类型之一：
   - 会员交易明细
   - 店内订单明细(已结账)
   - 收入优惠统计
   - 优惠券统计表
   - 会员储值消费分析表
   - 会员卡导出

2. 检查文件名是否包含上述关键词（不区分大小写）

3. 确认文件是 `.xlsx` 格式（不是 `.xls`）

4. 如果文件名不包含关键词，可以重命名文件添加关键词

**示例**：
- ✅ `山禾田_会员交易明细_2026-01-28.xlsx` - 可以识别
- ✅ `会员交易明细.xlsx` - 可以识别
- ❌ `data.xlsx` - 无法识别（缺少关键词）
- ❌ `data.xls` - 无法识别（格式不支持）

---

### 3. 处理 Excel 文件失败

**错误信息**：
```
处理 Excel 文件失败: xxx.xlsx。
可能原因：文件格式不正确、表头识别失败等。错误: ...
```

**原因**：
- 文件格式损坏
- 表头行不符合预期
- 文件结构异常

**解决方法**：
1. 尝试用 Excel 打开文件，确认文件可以正常打开
2. 检查表头行是否正确（参考 PRD 文档）
3. 如果文件损坏，尝试从源重新下载
4. 查看日志框中的详细错误信息

## 网络问题

### 1. 请求超时

**错误信息**：
```
REQUEST_TIMEOUT: ...
```

**原因**：
- 网络连接慢或不稳定
- 服务端响应慢
- 超时设置过短（默认 30 秒）

**解决方法**：
1. 检查网络连接状态
2. 确认服务端是否正常运行
3. 检查防火墙设置
4. 如果网络不稳定，等待网络恢复后重试
5. 系统会自动重试（2s/5s/10s 间隔）

**预防措施**：
- 在网络稳定的环境下使用
- 如果经常超时，可以考虑修改 `config.py` 中的 `request_timeout_seconds`

---

### 2. 连接错误

**错误信息**：
```
CONNECTION_ERROR: ...
```

**原因**：
- 网络连接中断
- DNS 解析失败
- 服务端不可达

**解决方法**：
1. 检查网络连接
2. 确认 webhook URL 是否正确
3. 检查 DNS 设置
4. 尝试 ping 服务端地址
5. 检查代理设置（如果使用代理）

---

### 3. HTTP 错误

**错误信息**：
```
HTTP_400: ...
HTTP_500: ...
```

**原因**：
- 请求格式错误（400）
- 服务端内部错误（500）
- 认证失败（401/403）

**解决方法**：
1. **HTTP 400**：检查请求数据格式，确认 platformKey 正确
2. **HTTP 500**：联系服务端管理员
3. **HTTP 401/403**：检查 platformKey 是否正确
4. 查看错误响应中的详细信息

---

### 4. 熔断器触发

**错误信息**：
```
已熔断: 会员交易明细 / 山禾田·日料小屋（龙华店）
触发熔断: ...
```

**原因**：
- 连续 5 个批次推送失败
- 网络或服务端持续异常

**解决方法**：
1. 点击"重置"按钮清除熔断状态
2. 检查网络和服务端状态
3. 确认 webhook URL 正确
4. 检查 platformKey 是否正确
5. 如果问题持续，联系技术支持

**注意**：
- 熔断器是为了防止无效请求刷屏
- 重置后可以继续推送，但需要先解决根本问题

## 数据库问题

### 1. 数据库锁定

**错误信息**：
```
database is locked
```

**原因**：
- 多个应用实例同时运行
- 数据库文件被其他进程占用
- SQLite WAL 文件异常

**解决方法**：
1. 确保只有一个应用实例在运行
2. 关闭所有应用窗口
3. 检查是否有残留进程：
   - Windows: 任务管理器
   - macOS/Linux: `ps aux | grep python`
4. 如果问题持续，删除以下文件后重新启动：
   - `siyu_etl.sqlite3`
   - `siyu_etl.sqlite3-wal`（如果存在）
   - `siyu_etl.sqlite3-shm`（如果存在）

**注意**：
- 删除数据库文件会丢失所有历史数据
- 建议先备份数据库文件

---

### 2. 数据库损坏

**错误信息**：
```
database disk image is malformed
```

**原因**：
- 数据库文件损坏
- 磁盘错误
- 异常关闭导致数据不一致

**解决方法**：
1. 尝试修复数据库：
   ```bash
   sqlite3 siyu_etl.sqlite3 "PRAGMA integrity_check;"
   ```
2. 如果修复失败，删除数据库文件重新开始
3. 检查磁盘是否有错误

**预防措施**：
- 正常关闭应用
- 避免强制终止进程
- 定期备份数据库文件

## 文件处理问题

### 1. 文件过大

**症状**：
- 处理速度慢
- 内存占用高
- 应用响应慢

**解决方法**：
1. 检查文件大小，如果超过 50MB，考虑分批处理
2. 关闭其他占用内存的程序
3. 如果文件行数超过 20 万行，可能需要调整 `constants.py` 中的 `MAX_SCAN_ROWS`

**限制**：
- 默认最大扫描行数：200,000 行
- 默认最大扫描列数：200 列

---

### 2. 表头识别失败

**症状**：
- 文件可以识别类型，但表头读取错误
- 数据列不对应

**解决方法**：
1. 检查表头行位置是否正确（参考 PRD 文档）
2. 确认表头行没有被合并单元格影响
3. 检查文件是否符合预期格式
4. 查看日志中的表头识别信息

## 配置问题

### 1. platformKey 未设置

**症状**：
- 推送失败
- HTTP 401/403 错误

**解决方法**：
1. 确认已设置环境变量 `SIYU_PLATFORM_KEY`
2. 检查环境变量是否正确：
   ```bash
   # Linux/macOS
   echo $SIYU_PLATFORM_KEY
   
   # Windows PowerShell
   echo $env:SIYU_PLATFORM_KEY
   ```
3. 如果未设置，按照 README 中的说明设置环境变量

---

### 2. Webhook URL 错误

**症状**：
- 连接失败
- HTTP 404 错误

**解决方法**：
1. 检查 `siyu_etl/config.py` 中的 webhook URL
2. 确认 URL 格式正确（以 `http://` 或 `https://` 开头）
3. 测试 URL 是否可以访问（使用浏览器或 curl）
4. 联系服务端管理员确认 URL

---

## 性能问题

### 1. 处理速度慢

**可能原因**：
- 文件过大
- 网络慢
- 数据库操作频繁

**优化建议**：
1. 分批处理大文件
2. 在网络稳定的环境下使用
3. 关闭不必要的后台程序
4. 如果数据库文件很大，考虑清理历史数据

---

### 2. 内存占用高

**可能原因**：
- 文件过大
- 数据库缓存
- 内存泄漏

**解决方法**：
1. 关闭其他占用内存的程序
2. 分批处理大文件
3. 重启应用释放内存
4. 如果问题持续，检查是否有内存泄漏
